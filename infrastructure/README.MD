
# Production Deployment

## how it works

```
┌─────────────────────────────────────────────────────────────────┐
│  GitHub Repository                                              │
│                                                                 │
│  ┌────────────────────┐   manual trigger  ┌──────────────────┐  │
│  │ deploy-infra.yml   │ ───────────────►  │  Bicep template  │  │
│  │ (workflow_dispatch)│                   │  main.bicep      │  │
│  └────────────────────┘                   └────────┬─────────┘  │
│          │                                         │            │
│          │  saves outputs as                       │ deploys    │
│          │  GitHub env variables                   ▼            │
│          │                                 ┌──────────────┐     │
│          ▼                                 │    Azure     │     │
│  ┌───────────────────┐                     │  (ACR, App   │     │
│  │ GitHub Environment│                     │  Service, FD,│     │
│  │  "production"     │                     │  SQL, etc.)  │     │
│  │                   │                     └──────────────┘     │
│  │  ACR_LOGIN_SERVER │                                          │
│  │  WEB_APP_NAME     │◄───── read vars ──────┐                  │
│  │  STORAGE_ACCOUNT  │                       │                  │
│  │  FD_PROFILE_NAME  │              ┌────────┴──────────┐       │
│  │  ...              │              │  CI/CD workflows  │       │
│  └───────────────────┘              │  (push to main)   │       │
│                                     │  backend-ci-cd    │       │
│                                     │  frontend-ci-cd   │       │
│                                     │  maintenance-ci-cd│       │
│                                     └───────────────────┘       │
└─────────────────────────────────────────────────────────────────┘
```

resource names (ACR, storage, web app, etc.) are auto-generated by Bicep using `uniqueString()`. after infrastructure deploys, the generated names are written back to GitHub Environment variables. the CI/CD workflows read those variables — no hardcoded resource names anywhere.

## prerequisites

- [Azure CLI](https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- [PowerShell Core (pwsh)](https://learn.microsoft.com/en-us/powershell/scripting/install/installing-powershell) — only needed for manual fallback
- an Azure subscription with Owner access
- a GitHub repository with Actions enabled

## one-time setup

the setup has four phases. phases 1-2 happen **before** deploying any Azure resources.
phase 3 is the deployment itself. phase 4 finishes configuration that depends on
resources created by Bicep.

```
Phase 1 ─ Azure prerequisites     create service principal + auth app registration
Phase 2 ─ GitHub configuration     environments, secrets, variables, PAT, .bicepparam files
Phase 3 ─ Deploy infrastructure    Bicep via GitHub Actions (or manual fallback)
Phase 4 ─ Post-deployment          seed Key Vault secrets, add FD redirect URI, upload images
```

---

### phase 1 — Azure prerequisites

follow the first two guides in `manual-setup/`, in order:

1. **[01-create-cicd-service-principal.md](manual-setup/01-create-cicd-service-principal.md)** — create the CI/CD App Registration + Service Principal in Entra ID, assign Contributor and User Access Administrator roles at subscription scope
2. **[02-create-auth-app-registration.md](manual-setup/02-create-auth-app-registration.md)** — create the OAuth 2.0 Auth App Registration, configure redirect URIs, enable ID tokens, and set the Application ID URI

each guide offers both **Azure Portal** and **Azure CLI** options — pick whichever you prefer and follow it end to end.

after completing these you'll have two separate Entra ID App Registrations:
- **CI/CD Service Principal** (guide 01) — with `clientId`, `clientSecret`, `tenantId`. used by GitHub Actions to authenticate with Azure and deploy resources. its credentials go into GitHub environment secrets as `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_TENANT_ID`
- **Auth App Registration** (guide 02) — with its own `clientId`, `clientSecret`, `tenantId`. used by the application for end-user sign-in via Easy Auth. its `clientId` and `tenantId` go into the `.bicepparam` files (hardcoded for production, environment secrets for arena). its `clientSecret` is stored in Key Vault in phase 4

> **Note:** The Key Vault doesn't exist yet — auth secrets are stored in Key Vault in phase 4,
> after Bicep creates the resources.

### phase 2 — GitHub configuration

#### step 1 — create GitHub Environments

go to your repo → **Settings** → **Environments** → **New environment**.

create three environments: `production` (live deployment), `arena-stress` (load testing), and `arena-security` (WAF/security testing). for each one, type the name exactly as shown, click **Configure environment**, and save. no protection rules are required.

#### step 2 — add GitHub secrets

go to **Settings** → **Secrets and variables** → **Actions** → **Environment secrets**.

there are two categories of secrets. the first four authenticate the **CI/CD Service Principal** (from [guide 01](manual-setup/01-create-cicd-service-principal.md)) — Azure CLI uses these to log in and run deployments. the remaining secrets are **deployment parameters** consumed by the Bicep template at deploy time.

**CI/CD authentication (all three environments):**

| Secret | Source | Description |
|--------|--------|-------------|
| `AZURE_CLIENT_ID` | guide 01 | CI/CD SP Application (client) ID |
| `AZURE_CLIENT_SECRET` | guide 01 | CI/CD SP client secret value |
| `AZURE_TENANT_ID` | Azure Portal → Entra ID → Overview | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Azure Portal → Subscriptions | Azure subscription ID |

all three environments can share the same SP credentials if the SP has subscription-level access. you only need separate SPs if you want environment isolation.

**Deployment parameters (all three environments):**

| Secret | Source | Description |
|--------|--------|-------------|
| `SQL_ADMIN_PASSWORD` | you choose | SQL Server admin password (consumed by every `.bicepparam` file via `readEnvironmentVariable()`) |

**Deployment parameters (arena environments only):**

| Secret | Source | Description |
|--------|--------|-------------|
| `ARENA_AUTH_CLIENT_ID` | guide 02 | **Auth App** Registration client ID — the app from [guide 02](manual-setup/02-create-auth-app-registration.md), **not** the CI/CD SP from guide 01 |
| `ARENA_AUTH_TENANT_ID` | guide 02 | **Auth App** Registration tenant ID |

> **Why arena externalises Auth App IDs but production hardcodes them:**
> Client IDs and tenant IDs are public identifiers (they appear in redirect URIs and OpenID discovery documents) — they are not secrets.
> Production hardcodes them directly in `main.bicepparam` because there is only one production Auth App and the values never change.
> Arena uses `readEnvironmentVariable()` because arena deployments may use a **different** Auth App Registration than production (e.g. with different redirect URIs or a separate tenant), so the values are parameterised.

#### step 3 — add GitHub variables (manual, per environment)

go to **Settings** → **Secrets and variables** → **Actions** → **Variables** tab → select an environment.

add two variables manually for each environment:

- `RG_NAME` — the resource group name, e.g. `TaskManager-RG` (created automatically if it doesn't exist)
- `LOCATION` — the Azure region, e.g. `italynorth`

the remaining six variables (`ACR_LOGIN_SERVER`, `WEB_APP_NAME`, `STORAGE_ACCOUNT_NAME`, `FD_PROFILE_NAME`, `FD_ENDPOINT_NAME`, `FD_ENDPOINT_HOSTNAME`) are auto-populated by the `deploy-infra` workflow after Bicep deploys. don't set them manually.

#### step 4 — create a Personal Access Token (PAT)

the `deploy-infra` workflow needs a PAT to write environment variables back to GitHub (the default `GITHUB_TOKEN` doesn't have permission to manage environment variables).

1. Go to **GitHub.com** → click your **profile picture** → **Settings** → **Developer settings** → **Personal access tokens** → **Tokens (classic)**
2. Click **Generate new token (classic)**
3. Name it `deploy-infra-pat` (or similar)
4. Set expiration (90 days is reasonable; adjust it to your rotation needs)
5. Select the **`repo`** scope (full control of private repositories)
6. Click **Generate token** and **copy the value immediately**
7. Go back to **repo Settings** → **Secrets and variables** → **Actions** → **Repository secrets**
8. Click **New repository secret**, name it `GH_PAT`, paste the token value

the PAT is a repository secret (not environment secret) because it needs to write to whichever environment is selected at runtime.

#### step 5 — prepare Bicep parameter files

```bash
# Copy the example files and fill in real values
cp infrastructure/bicep/main.bicepparam.example         infrastructure/bicep/main.bicepparam
cp infrastructure/bicep/main-arena.bicepparam.example    infrastructure/bicep/main-arena-stress.bicepparam
cp infrastructure/bicep/main-arena.bicepparam.example    infrastructure/bicep/main-arena-security.bicepparam
```

edit each `.bicepparam` file. the files are safe to commit — all actual secrets are read at deploy time via `readEnvironmentVariable()`, never written into the file.

understanding what is in each file:

| Value | Production | Arena (stress / security) | Sensitivity |
|-------|-----------|--------------------------|-------------|
| `location`, `projectName`, `tags` | hardcoded | hardcoded | not sensitive |
| `sqlAdminLogin`, `sqlDatabaseName` | hardcoded | hardcoded | not sensitive (login name, not password) |
| `sqlAdminPassword` | `readEnvironmentVariable('SQL_ADMIN_PASSWORD')` | `readEnvironmentVariable('SQL_ADMIN_PASSWORD')` | **secret** — set as GitHub environment secret (step 2) |
| `authAppClientId` | hardcoded (from [guide 02](manual-setup/02-create-auth-app-registration.md)) | `readEnvironmentVariable('ARENA_AUTH_CLIENT_ID')` | not sensitive — public identifier |
| `authAppTenantId` | hardcoded (from [guide 02](manual-setup/02-create-auth-app-registration.md)) | `readEnvironmentVariable('ARENA_AUTH_TENANT_ID')` | not sensitive — public identifier |
| Spring profile overrides, WAF params | — | hardcoded | not sensitive |

> **Two different App Registrations — do not mix them up.**
> The `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` secrets (step 2) belong to the **CI/CD Service Principal** from [guide 01](manual-setup/01-create-cicd-service-principal.md) — they authenticate the deployment pipeline.
> The `authAppClientId` / `authAppTenantId` values in the `.bicepparam` files belong to the **Auth App Registration** from [guide 02](manual-setup/02-create-auth-app-registration.md) — they configure Easy Auth & OAuth2 so end users can sign in.
> These are two separate Entra ID App Registrations with different purposes and different credentials.

> **Manual fallback note:** when deploying from your machine instead of GitHub Actions, the `readEnvironmentVariable()` calls read from OS environment variables. set `SQL_ADMIN_PASSWORD` (all environments) and `ARENA_AUTH_CLIENT_ID` + `ARENA_AUTH_TENANT_ID` (arena only) before running the PowerShell script.

### phase 3 — deploy infrastructure

#### option A — GitHub Actions (recommended)

##### first deployment

1. commit and push your `.bicepparam` files to GitHub (they contain no secrets)
2. go to **Actions** → **deploy-infra** → **Run workflow**
3. select environment `production`, mode `what-if`
4. review the output — it shows what Azure would create, modify, or delete
5. if it looks right, run again with mode `deploy`
6. after it finishes, check the workflow summary — it shows the six auto-populated variables
7. verify in **Settings** → **Environments** → `production` → **Variables** — all six should be there

##### updating infrastructure

whenever you change `main.bicep` or the `.bicepparam` files, run `deploy-infra` with `what-if` first to preview, then with `deploy` to apply. the environment variables update automatically if resource names change.

#### option B — manual fallback (PowerShell)

if GitHub Actions is unavailable or you prefer deploying from your machine:

```bash
cp infrastructure/ps1-az-scripts/.env.example infrastructure/ps1-az-scripts/.env
# Edit .env with your SP credentials, RG name, location, SQL password
```

```powershell
# preview what Azure would create/modify/delete
pwsh -File infrastructure/ps1-az-scripts/main.ps1 -WhatIf

# deploy for real
pwsh -File infrastructure/ps1-az-scripts/main.ps1
```

both commands load `.env`, log in with the SP, create the RG (if needed), and deploy Bicep.

after a manual deploy, copy the output values and manually set the six GitHub environment variables so the CI/CD workflows work. the outputs are printed at the end of deployment.

---

### phase 4 — post-deployment

after Bicep creates the Azure resources, complete the remaining configuration:

**[03-post-deployment-setup.md](manual-setup/03-post-deployment-setup.md)** covers three steps:

1. **Seed Key Vault secrets** — add the 7 manual secrets (Auth App client ID / client secret / tenant ID from [guide 02](manual-setup/02-create-auth-app-registration.md), redirect URI, admin email, JWT secret, ACS sender address). Bicep auto-populates the other 10
2. **Add Front Door redirect URI** — add the FD endpoint hostname (`https://<fd-endpoint>.azurefd.net/auth/callback`) to the Auth App Registration's redirect URIs
3. **Upload default images** — upload the `default-images/` folder to Blob Storage

> **Important:** The backend App Service reads Key Vault secrets at startup. If any manual secret
> is missing, the app fails to start. The `MICROSOFT_PROVIDER_AUTHENTICATION_SECRET` app setting
> (Easy Auth) also resolves via Key Vault — it must exist before Front Door's origin authentication works.

after completing phase 4, the system is fully operational.

## application deployments (automatic)

once infrastructure is deployed and environment variables are populated, the CI/CD workflows trigger automatically on push to `main`:

- **backend-ci-cd** triggers on backend or contracts path changes. builds and tests the Spring Boot app, pushes a Docker image to ACR, and updates the App Service container.
- **frontend-ci-cd** triggers on frontend path changes. builds the React app, uploads static files to blob storage, and purges the Front Door cache.
- **maintenance-ci-cd** triggers on maintenance path changes. builds the maintenance jar, pushes to ACR, and updates both ACA cron jobs.

all three use `environment: production` and read resource names from `vars.*`.

## design decisions

**`uniqueString()` for resource names** — Azure requires globally unique names for storage accounts, ACRs, web apps, etc. instead of picking names manually and hoping they don't collide, Bicep generates deterministic names from `uniqueString(resourceGroup().id)`. same RG always produces the same names (idempotent). different RG produces different names (no collisions between environments).

**subscription-level SP roles** — resource groups come and go (redeploy, rename, tear down). assigning roles at RG level breaks every time the RG is recreated. the subscription persists, so assigning Contributor + User Access Administrator once at subscription scope avoids that issue. the App Registration + SP cost nothing when idle.

**PAT for deploy-infra** — the default `GITHUB_TOKEN` can't create or update GitHub environment variables. a classic PAT with `repo` scope is the simplest solution. fine-grained PATs with `Variables: Read and Write` + `Environments: Read and Write` permissions also work if you prefer narrower scope.

**`--auth-mode login` instead of storage account keys** — the CI/CD workflows authenticate to blob storage using the SP's Azure identity rather than a storage account key. this eliminates the `STORAGE_ACCOUNT_KEY` secret and relies on RBAC instead (the SP gets `Storage Blob Data Contributor` via a Bicep role assignment).
